# Contextifier v0.2.5 — 핸들러 품질 심층 분석

> 분석 일자: 2025-04-01
> 분석 대상: 17개 핸들러 (15개 디렉토리, ~130파일, ~19,000 LOC)

---

## 1. 핸들러 분류 체계

### 1.1 포맷 분류

| 분류 | 핸들러 | 공통 특성 |
|------|--------|----------|
| **모던 OOXML** | docx, pptx, xlsx, hwpx | ZIP/XML 구조, 풍부한 파서 라이브러리 |
| **레거시 바이너리** | doc, ppt, xls, hwp | OLE2/BIFF 구조, 제한된 파서 라이브러리 |
| **PDF** | pdf, pdf_default, pdf_plus | 복합 구조, 다양한 추출 전략 필요 |
| **리치 텍스트** | rtf, html | 마크업 기반, 파싱 복잡도 높음 |
| **플레인 데이터** | csv, tsv, text | 단순 구조, 인코딩이 주요 이슈 |
| **미디어** | image | OCR 의존, 메타데이터 추출 |

### 1.2 위임 관계

```
doc  ──[실제 DOCX ZIP]──▶ docx
doc  ──[실제 RTF 내용]──▶ rtf
ppt  ──[실제 PPTX ZIP]──▶ pptx
xls  ──[실제 XLSX ZIP]──▶ xlsx
hwp  ──[실제 HWPX ZIP]──▶ hwpx
pdf  ──[모드 선택]──────▶ pdf_default / pdf_plus
```

### 2.1 PDF 계열

#### PDF Router (`handlers/pdf/`)

- **평가**: ✅ 설계 양호, 라우팅 로직 명확

#### PDF Default (`handlers/pdf_default/`)
- **파일**: content_extractor.py (282)
- **기술**: PyMuPDF (`fitz`) 직접 추출
- **기능 매트릭스**:

| 기능 | 구현 상태 | 품질 | 상세 |
|------|----------|------|------|
| 텍스트 추출 | ✓ | 양호 | `page.get_text("text")` |
| 테이블 감지 | ✓ | 보통 | `page.find_tables()`, TABLE_SIZE=50 하드코딩 |
| 이미지 추출 | ✓ | 양호 | xref 기반 추출, 중복 제거 |
| 차트 추출 | ✗ | - | 미구현 |
| OCR 폴백 | ✗ | - | **🔴 치명적: needs_ocr 감지 후 무시** |
| 메타데이터 | ✓ | 양호 | PDF 메타데이터 표준 필드 |

**🔴 Critical Issue — Scanned PDF 데이터 손실**:
```python
# preprocessor.py에서 스캔 감지 완료
needs_ocr = self._detect_scan(doc, page_count)  # ✓ True 반환

# content_extractor.py에서 이 플래그 완전 무시
# → 스캔 PDF에서 텍스트 0바이트 반환, 페이지 태그만 출력
```
- **실제 영향**: 실무 PDF의 ~30%가 스캔 문서 (보험, 계약서, 정부문서)
- **필요 조치**: OCRProcessor 연동 또는 Tesseract 직접 호출

### 2.3 Office 레거시 (OLE2/BIFF)

#### DOC (`handlers/doc/`)
- **파일**: 6개 (~800 LOC)
- **기술**: olefile + 휴리스틱 텍스트 스캐닝

| 기능 | 구현 상태 | 품질 | 상세 |
|------|----------|------|------|
| 텍스트 추출 | ~ | 보통 | **UTF-16LE 바이트 스캐닝** (휴리스틱) |
| 테이블 추출 | ✗ | - | **🔴 미구현** (FIB 파싱 필요) |
| 이미지 추출 | ~ | 부분 | OLE 스트림 추출 |
| 차트 추출 | ✗ | - | 미구현 |
| 위임 | ✓ | 양호 | ZIP 헤더 → DOCX, RTF 시그니처 → RTF로 위임 |

**하드코딩된 임계값**:
```python
MIN_TEXT_FRAGMENT_LENGTH = 20    # 텍스트 조각 최소 길이
MIN_UNICODE_BYTES = 4           # 유니코드 최소 바이트
CJK_HIGH_BYTE_RANGES = [...]    # CJK 바이트 범위 (하드코딩)
```

**🔴 Critical Issues**:
1. **텍스트 추출이 휴리스틱**: 실제 Word 구조(Piece Table, FIB)를 파싱하지 않음
2. **테이블 완전 미지원**: 복잡한 DOC 문서의 테이블 데이터 100% 손실
3. **헤더/푸터/필드/수식 누락 가능**: 바이트 스캔만으로는 문서 구조를 재현할 수 없음

**권장 조치**: OLE2 바이너리 규격 기반 네이티브 파싱 고도화
- FIB(File Information Block) 파싱 → Piece Table 기반 정확한 텍스트 추출
- Table Stream(0Table/1Table) 구조 파싱 → 테이블 데이터 복원
- Data 스트림 내 임베디드 객체 추출 강화

## 5. 네이티브 바이너리 파싱 고도화 전략

레거시 포맷(DOC, PPT, XLS)의 기능 갭은 **바이너리 규격 기반 네이티브 파싱**으로 해결한다.
외부 도구(LibreOffice 등)에 의존하는 변환 경로는 불필요한 시스템 의존성을 생성하고 파이프라인 구조를 훼손하므로 채택하지 않는다.

```
DOC  ──[olefile+FIB 파싱]────▶ 텍스트(Piece Table) + 테이블(Table Stream) + 이미지(OLE 스트림)
PPT  ──[olefile+레코드 파싱]──▶ 텍스트(Atom) + 테이블(Table Record) + 이미지(Pictures 스트림)
XLS  ──[xlrd+olefile 보완]───▶ 텍스트/테이블(xlrd) + 이미지(OLE 스트림) + 차트(BIFF 레코드)
```

**핵심 원칙**:
1. 모든 문서는 해당 포맷의 공개 바이너리 규격에 따라 직접 파싱한다
2. OLE2 Compound Document, BIFF, PowerPoint Binary 등 명확한 규칙에 의해서만 처리한다
3. 외부 변환 도구를 통한 포맷 변환(DOC→DOCX 등)은 사용하지 않는다
4. 기존 올바르게 구현된 파싱 패턴(DOC 이미지 스캔, PPT Pictures 스트림)을 재활용한다

**제거 대상**: `services/libreoffice.py` (133 LOC) — 외부 도구 의존 모듈 삭제
